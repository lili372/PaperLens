<h1 align="center">PaperLens</h1>
<p align="center">基于论文证据的科研选题推荐 Agent</p>

<p align="center">
输入一个研究领域，系统自动检索论文、抽取方法模块、生成候选选题，并通过 RAG 证据查证与可做性审查过滤“看起来相关但实际做不起来”的研究方向。
</p>

## 为什么做这个项目

科研选题早期的核心困难不是“找不到论文”，而是：

- 不知道哪些论文应该优先精读；
- 不知道一个方法模块能否迁移到另一个问题；
- 很容易把表面相关的论文拼成一个实际上不可做的选题。

PaperLens 的目标不是替代人工精读，而是帮助用户在精读前完成第一轮判断：**哪些方向值得继续看，哪些组合需要收窄，哪些想法应该暂缓或放弃。**

## 核心流程

```text
输入研究领域
  -> 生成检索计划
  -> arXiv 检索与 PDF 下载
  -> 论文筛选与人工确认
  -> 论文画像抽取
  -> 候选研究方向生成
  -> RAG 证据查证
  -> 可做性审查与方向修订
  -> 输出研究方向报告
```

## 一次推荐会长什么样

```text
推荐方向：
用 ProPML 替代 POMDP Stage 1 判别头损失中的 BCE 项，保留图正则化项，在小实验上验证兼容性。

建议先读：
2604.09064v1、2507.05992v2

证据判断：
base_problem: supported
module_method: supported
connection: partially_supported

主要风险：
ProPML 的概率损失与 POMDP reward / loss 之间存在兼容性风险，需要先做小规模实验验证。
```

系统不会只给一个方向标题，而是同时说明这个方向来自哪些论文、哪些部分已有证据支持、哪里还需要进一步验证，以及应该先读哪几篇论文。

## 它如何避免假选题

科研选题推荐最容易出问题的地方，是把几篇表面相关的论文硬拼成一个方向。PaperLens 没有直接让模型从论文列表生成最终答案，而是把过程拆成几个可检查的步骤：先找论文，再抽取可迁移的方法模块，再生成候选方向，最后回到论文证据中核查。

### 回到论文里查证

每个候选方向不会直接进入最终报告，而是先拆成三类问题：

- 目标问题是否真实存在；
- 被迁移的方法模块是否真实成立；
- 这个方法模块和目标问题之间是否真的能连接起来。

系统会基于 PDF RAG 召回原文证据，再判断这些问题是有支持、部分支持、缺少证据，还是存在明显反证。

### 对风险方向做分流

有些方向不是完全不能做，而是需要收窄问题、保留原方法结构，或者先做小规模实验验证。因此系统不会只做“推荐 / 不推荐”的二分判断，而是把方向分成几类：

- 证据比较充分的方向，进入主推荐；
- 有价值但需要收窄的方向，修改后保留；
- 材料不足的方向，暂缓推荐；
- 存在明显反证或依赖冲突的方向，直接拒绝。

### 用失败案例反推流程改进

项目试跑中出现过“同领域论文被误组合”“理论贡献被误当作可迁移模块”等问题。后续迭代时，我把这些失败案例反推回论文画像和候选生成环节，限制系统只抽取可迁移、可替换的方法模块，并强化目标问题和方法模块之间的匹配约束。

## 功能概览

- **研究方向推荐**：从领域输入出发，生成候选选题、证据依据、风险提示和建议精读论文。
- **综述分析**：输入研究方向，自动检索 arXiv 论文并生成带来源标注的导航型综述。
- **人工确认节点**：在检索计划、召回论文和进入分析的论文范围等关键节点允许用户确认。
- **Markdown 导出**：研究方向报告和综述报告均支持导出。

## 技术栈

- 后端：Python、FastAPI、AutoGen、ChromaDB、arXiv API
- 前端：Vue 3、Vite、SSE
- 模型接口：DeepSeek 兼容接口、DashScope Embedding
